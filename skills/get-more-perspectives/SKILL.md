# Get More Perspectives

在拍板之前,先把一个决策放到 3-5 个**真正不同的视角**下照一遍。

这个 skill 存在的意义,是对抗决策里最常见、也最贵的失败:**隧道视野**。当一个人(或一个 AI)盯着一个决策看久了,ta 会不自觉地收敛到某一个框架、某一套价值排序、某一个已经隐隐偏向的答案,然后为它找理由,而不是真的去看别的可能。等到后悔的时候,往往会发现"其实当时该考虑的那个角度,一开始就在,只是没人替我说出来"。

本 skill 的工作**不是替用户做决定**,而是把决策的选项空间尽量铺开、把不同视角之间真正的分歧和它们各自在优化什么看清楚,最后交给用户一张标好岔路口的"决策地图"——**判断权始终在用户手里**。

## 唯一真正重要的事:视角要真的不一样

"多找几个角度"最容易翻车的地方,是产出 5 个**同一个观点的不同措辞**:都在优化同一个东西、都默认同一套前提、只是换了几个词。这种"伪多样性"比只给一个观点更糟,因为它制造了"我已经全面考虑过了"的错觉。

一个视角之所以**算一个新视角**,是因为它在下面至少一个维度上和别的视角真的不同:

- **它在优化什么**——速度 / 质量 / 成本 / 可逆性 / 简单 / 长期可选性,不同视角优先级不一样,结论才会分叉。
- **它代表谁的利益**——用户本人 / 维护它的人 / 接手它的人 / 客户 / 团队 / 一年后的自己。
- **它站在哪个时间尺度**——这周最优、一年后最优、五年后最优,常常是三个不同答案。
- **它用什么心智模型**——工程师 / 经济学家 / 设计师 / 律师 / 历史学家……不同学科的框架会照出决策里被忽略的那一维。
- **它的性格**——务实派(什么能落地?)、怀疑派/红队(什么会让它崩?)、纯粹派(什么才是对的?)、逆向派(如果前提本身就是错的呢?)。

判断标准很简单:**如果两个视角能得出同一个建议、给出同一套理由,它们就只算一个视角。** 生成时始终盯住"它们的结论会不会真的打架"——会打架,才有价值。

## 工作流

### 第 1 步:把决策框清楚

视角的质量取决于问题被框得有多准。用户经常只丢一句"我该选 A 还是 B",但缺了关键信息,盲目开跑只会得到一堆空对空的意见。动手前先确认这几件事(用户已经说清楚的别重复问):

1. **到底在决定什么**——一句话把决策收敛成一个明确的问题。
2. **候选选项有哪些**——如果用户只给了一两个,先想想是不是还有没被提到的第三条路(很多时候最好的视角就是"你以为只有 A/B,其实还有 C")。
3. **代价和可逆性**——这个决定错了要付多大代价、能不能撤回。这决定了要不要大动干戈找 5 个视角,还是点到为止。
4. **用户真正在乎什么**——ta 嘴上说的目标,和 ta 真正的约束(时间、钱、风险承受度、别人的看法)可能不是一回事。

信息缺得多就**一次性列出来问**,别挤牙膏;缺得少、或用户说"你先跑",就有多少用多少,在产出里把假设标出来(`(假设)`),并说明假设变了结论会怎么变。

### 第 2 步:为这个决策挑 3-5 个会真正分叉的视角

**不要套固定模板。** 好的视角组合是为**这个具体决策**量身选的——照出它特有的张力。用上面那个"维度菜单"来选,而不是每次都用同一批角色。

选择时把握三条:

- **覆盖真正的张力轴**。先想清楚这个决策里最主要的权衡是什么(比如"快但脆" vs "慢但稳"),让视角分别站在张力轴的两端,而不是都挤在中间。
- **一定要有一个视角在唱反调**。至少放一个红队/怀疑派视角,专门去攻击当前看起来最顺眼的那个选项——"这个方案会怎么崩?我们在骗自己什么?"。**决策里最大的价值往往来自这一票**,因为它抓的正是你正准备犯的错。
- **数量随代价走**。低代价/可逆的决策,2-3 个视角够了;高代价/难撤回的,值得上到 4-5 个,并把红队视角做厚。

### 第 3 步:生成视角(优先用独立子代理)

**独立性是关键**——如果几个视角互相看得见对方的推理,它们会不自觉地互相锚定、抹平分歧,又退化回"伪多样性"。所以:

- **有子代理时**:每个视角开一个独立的 subagent 并行跑,彼此不共享上下文。给每个 subagent 的指令是:**完全代入这个视角**,像一个诚实的该领域专家那样,把这个立场论证到最强,并明确说出它在优化什么、愿意牺牲什么、什么证据会让它改变主意。
- **没有子代理时**(比如 Claude.ai、或本身已在一个 subagent 里、或决策小到不值得开 5 个 agent):在**同一个对话里依次**扮演每个视角,但每切换一个视角就**主动"清零"一次**——提醒自己放下上一个视角的结论,重新从这个视角的价值观出发。不要让上一个视角的答案带节奏。

每个视角用统一的结构产出,这样才可比:

- **主张**——这个视角会怎么选、会做什么。
- **为什么**——驱动它的那个价值观 / 前提假设。
- **它在优化什么、愿意放弃什么**——把隐含的取舍摆到台面上。
- **最强的一条理由或证据**。
- **它的软肋 / 什么会让它改变主意**——诚实地说出这个视角在哪里最站不住脚。

### 第 4 步:铺成决策地图,但不替用户下判断

把 3-5 个视角收拢成一张能帮人决策的地图。**这一步不给"我建议选 X"**,而是把用户自己拍板需要的东西交到 ta 手上:

- **共识区**——哪些点几个视角都同意?这是最稳的地基,可以放心押注。
- **真正的分歧,以及分歧的根源**——哪些点视角之间真的打架?更重要的是**为什么打架**:分歧几乎总能追溯到"价值排序不同"或"某个事实还不确定"这两类。点破根源,分歧才可决。
- **岔路口(crux)**——用最少的几个问题,把决策拆成清清楚楚的岔路:"**如果你更看重 X 而不是 Y,那么视角 A 那条路更合理;反过来则指向视角 C。**" 这是"不替你决定"的正确做法:不替用户选,而是把选择的岔路口标注清楚,交到 ta 手里。
- **意外收获**——有没有哪个视角照出了一开始的框架里根本没有、但很关键的东西?一个天真的做法会漏掉什么?

## 输出模板

按下面的结构输出 Markdown,**用用户所用的语言**(用户说中文就中文,英文就英文)。视角在前,决策地图在后。

```markdown
# [决策一句话] —— 多视角审议

## 决策快照
把决策、候选选项、代价/可逆性、用户真正在乎的东西,用几行交代清楚。
有信息缺口的用 `(假设)` 标出。

## 视角一:[给它起个能一眼看懂立场的名字,如"红队 / 求稳派"]
- **主张**:……
- **为什么**:……
- **在优化 / 愿意放弃**:……
- **最强理由**:……
- **软肋**:……

## 视角二:……
（同上结构,3-5 个）

## 决策地图
- **共识区**:……
- **真正的分歧 & 根源**:……
- **岔路口**:如果你更看重 ___,选 ___;如果更看重 ___,选 ___。
- **一开始容易漏掉的**:……
```

## 用几个 / 什么时候别用

- **别硬凑数**。视角是"真的不同"才有意义。如果这个决策真的只有 2 个诚实的立场,给 2 个扎实的,好过灌水到 5 个。
- **代价小就轻量跑**。可逆、改起来不疼的决定,别开 5 个子代理去审议一个变量命名——那是浪费。这个 skill 是给"改起来有代价、又没有显然答案"的决策用的。
- **用户已经决定了就别再铺**。如果 ta 其实只是想让你去执行,别用一堆视角拖住 ta;顶多提一句"有个角度你可能想快速看一眼",然后照办。
