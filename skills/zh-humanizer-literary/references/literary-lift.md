# 文采增强

当草稿已经不明显 AI，但仍然发扁、无记忆点或过于功能化时，读取这个文件。

目标是增加表达力量，不是模仿任何人。不要命名或模仿具体公众人物。只提取通用写作机制：命名、可见意象、节奏和动作。

## 核心机制

好的文采增强做四件事：

1. **命名处境**：把模糊问题压成读者能记住的一句话。
2. **让不可见的东西可见**：把时间、愿望、混乱、成长、价值或信任变成可被看见的东西。
3. **给句子一个容器**：一页纸、格子、清单、产品界面、评论、offer、会议、屏幕、笔记、表格或一段周期。
4. **落回动作或证据**：能被写下、测试、观察、重复或复盘。

不要停在漂亮句子。好句子应该帮读者做事或看清楚事。

## 命名动作

草稿有洞察但没有钩子时，用命名。

- 把情绪变成可处理对象：焦虑 -> 一页要写的纸、一个要命名的决定、一个要排进日程的动作。
- 把抽象工作变成可见任务：内容策略 -> 让受众正确复述产品。
- 把价值变成读者测试：读者看完后能说出什么？
- 把重复失败变成系统缺口：不是「不自律」，而是「没容器、没反馈、没返回路径」。
- 只有当状态词能被管理时，才把它命名成短语。

好的命名短、具体、能用。坏命名像课程包装。

## 可见意象

优先使用靠近真实工作的意象：

- 纸、页面、边距、下划线、括号、笔记
- 屏幕、产品页、评论区、搜索框、offer、咨询
- 表格、清单、格子、序列、版本
- 房间、桌面、工作台、货架、证据墙
- 路径、门槛、第一步、返回点

每个段落只用一个主意象。不要把纸、太阳、建筑、运动员、火和轨道堆在一起。

## 状态转对象

有帮助时，把形容词、状态或价值变成可管理对象：

```text
状态/价值 -> 命名对象 -> 容器 -> 动作/证据
```

示例：

- 「清晰」-> 可复述的一句话 -> 产品页 -> 用户能复述
- 「信任」-> 证据货架 -> 案例笔记/评论/截图 -> 读者看见证明
- 「好内容」-> 一个可见判断 -> 笔记结构 -> 收藏或咨询提升
- 「稳定写作」-> 每周页面 -> 固定时间格 -> 周五前四个草稿

规则：

- 命名对象必须指向真实容器或动作。
- 不要每句话都造品牌词。
- 保留一个强命名。命名太多会显得制造感很重。

## 节奏

用节奏制造记忆：

- 先给清楚判断。
- 接一句较长的句子，展示场景或证据。
- 用短句制造转弯。
- 结尾落在动作、后果或安静意象上。

可用句式：

```text
你不是缺 X。你缺的是 Y。
先让 Y 看得见。
再决定保留什么、删掉什么、测试什么。
```

```text
这不是口号问题。
这是翻译问题：东西有价值，但读者复述不出来。
```

```text
不要让文字更大声。
让证据站得更近。
```

## 有力量但不鸡汤

文采可以温暖、有力，但必须落地。

优先：

- 「先把这句话写到纸上，看看它能不能被读者复述。」
- 「不要让漂亮词替你工作。证据要站到句子前面。」
- 「一段内容真正变好，是读者突然知道自己下一步该看哪里。」

避免：

- 普遍承诺
- 英雄式自我改造
- 空洞鼓励
- 口号堆叠
- 过多感叹号
- 没有下一步动作的「燃」

## 按内容形态应用

### 社交内容

加一个被命名的矛盾、一个可见场景、一个可记住的句子和一个动作/检查。

### 个人随笔

原文有第一人称判断时保留。让读者看见想法是怎么形成的：工作现场、失败草稿、产品细节、重复观察。

### 诊断报告

谨慎使用文采。报告优先清晰。可以加一个有力章节开头或可记住的区分，然后回到证据和建议。

### 提案 / 服务文案

命名对方卡点和解决路径。除非服务真的处理人生层面的变化，否则不要写得过度戏剧化。

### 产品文案

让用户动作更可见。不要把界面文案写成诗。

## 最终检查

输出文采增强版前，问自己：

- 我增加的是记忆点，还是装饰？
- 事实安全吗？
- 我是不是只用了一个主意象？
- 有没有落到动作、测试或证据？
- 这仍然像一个有经验的人在说话，而不是鸡汤账号吗？
