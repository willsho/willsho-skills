# Transcript to Article（访谈深度解读）

## 任务定义

把一段原始访谈/播客 transcript 转化为中文深度解读文章。目标不是翻译，而是**重新提炼**：提取真正有价值的观点，补充背景知识，用中文读者更熟悉的表达方式重新呈现。

输出的读者预期：关心这个话题的中文读者，没有时间看完整个访谈，但需要理解关键观点和背后的含义。

---

## 输出格式

### 文件结构

输出为 Markdown 文件，文件名格式：`{嘉宾名} 最新访谈：{一句话核心论点}.md`

例如：
- `Karpathy 最新访谈：Vibe Coding 只是开始，真正重要的是 Agentic Engineering.md`
- `Cat Wu 面试了几百个 PM 候选人，几乎没人答对一个问题：AI 产品经理到底应该干什么？.md`

标题要有信息量，不要用"访谈整理"、"播客回顾"这类泛标题，要直接点明嘉宾的核心主张或最反直觉的观点。

### YAML frontmatter

```yaml
---
title: "完整文章标题"
source: "原始来源 URL（如果用户提供了的话）"
author:
  - "[[作者/整理人名]]"
published: YYYY-MM-DD
created: YYYY-MM-DD
description: "一句话摘要，点明核心价值"
tags:
  - "clippings"
---
```

### 正文结构（按顺序）

**1. 开场段落（3-5句）**

不加标题，直接描述：
- 这个人是谁（1句，挑最有力的身份标签）
- 这次访谈发生在哪、谈了什么（1句）
- 最令人意外或最有价值的洞察（1-2句，吸引读者继续看）

**2. 要点速览**

用 `## 要点速览` 标题，以 bullet list 形式列出 6-10 条精华观点，每条 20-60 字。要求：
- 每条是一个完整论断，不是话题标签
- 重要概念用**加粗**
- 按重要性排序，最有洞察力的放最前面

**3. 分节正文**

用 `## 1. 标题` 或 `## 【1】标题` 格式，每节覆盖访谈中的一个核心话题。每节包含：

- **问题转述**（可选）：如果是问答格式，用 "嘉宾被问到 XX" 或 "主持人问：XX" 开始，但要精炼，不要照搬原话
- **核心观点**：用自己的语言重新表达嘉宾的主张，要比原文更清晰、更有力
- **加粗核心句**：每节至少有一句加粗的"金句级"观点
- **直接引语**（重要的话）：用 `>` 引用中文翻译，后面括号里跟英文原文
- **编者注**（如需要）：对专业术语、人物背景、事件时间线做扩充。格式：`> **编者注：** ...`

节数量建议：8-20节，根据 transcript 长度决定。不要把每个问题都拆成一节，合并相关内容，按主题而非按原始顺序组织。

**4. Q&A 速览**

用 `## Q&A 速览` 标题，列出 3-8 个访谈中出现的具体问答，格式：
```
**问：XX？** 答：YY。
```

挑选访谈中最有价值或最典型的 Q&A，不必覆盖全部。

**5. 编辑手记**

用 `## 编辑手记：N 个值得关注的矛盾` 或类似标题，写 1-4 段，指出：
- 嘉宾说法前后的矛盾
- 嘉宾回避的尖锐问题
- 嘉宾主张的隐含前提
- 值得跟踪的后续信号

这部分是整篇文章最有价值的部分，代表编辑的独立判断。不要只是总结，要有真正的批判性分析。

---

## 写作要求

### 不是翻译，是提炼

原始 transcript 通常包含大量重复、废话、口头禅、赞助商广告等。这些要全部剔除。每一段都要经过提炼，保证信息密度。

判断标准：如果一段话删掉，读者不会损失任何信息，那就删掉。

### 补充背景，但要区分

文章服务中文读者。原文中提到的人名、公司、术语，如果中文读者可能不熟悉，在**编者注**里扩充。但编者注里的信息要明确标注来源是编辑补充，不要混入正文让读者误以为是嘉宾说的。

### 批判性保持独立

不要因为嘉宾是大佬就变成粉丝文章。如果嘉宾回避了某个问题、说法前后矛盾、或者给出了一个方便的解释但忽略了真正的争议，要在编辑手记里点出来。

### 引语的处理

- 重要的原话：先写中文翻译，括号里附英文原文
- 格式：`> 中文翻译（"English original"）`
- 翻译要准确但不要死板，尽量保持原话的语气和力度

### 专业术语

涉及技术术语、产品名称、行业词汇时，第一次出现加「注释」或在编者注里解释。对英文缩写，第一次出现时写全称，括号里写缩写。

---

## 处理步骤

1. **通读全文**：快速浏览整个 transcript，识别：谁是嘉宾、谁是主持人、访谈的核心话题是什么
2. **提取骨架**：列出 transcript 中的主要话题和嘉宾的核心论点（内部思考，不写进输出）
3. **确定标题**：找到最有力的、最能代表嘉宾核心主张的一句话作为副标题
4. **写要点速览**：用 6-10 条精华 bullet 帮读者决定是否值得细读
5. **按主题写正文**：不按原文顺序，按逻辑和重要性重新组织
6. **写编辑手记**：最后写，最需要独立判断

## 参考示例

以下是两个标准输出示例，用于理解目标风格：

**示例 1 标题风格：**
`Cat Wu 面试了几百个 PM 候选人，几乎没人答对一个问题：AI 产品经理到底应该干什么？`

开场段落风格：
> Cat Wu 是 Anthropic Claude Code 和 Cowork 的产品负责人，和 Boris Cherny 搭档，带着团队**把产品功能的交付周期从半年压到了一天**。在 Lenny's Podcast 最新一期中，Cat 聊了 Anthropic 内部的速度文化、PM 角色的剧变...

**示例 2 标题风格：**
`Karpathy 最新访谈：Vibe Coding 只是开始，真正重要的是 Agentic Engineering`

引语格式示例：
> **我记不得上一次我需要纠正它是什么时候了。然后我就越来越信任这个系统。**

编者注格式示例：
> **编者注：** Vibe Coding 不适合硬译成"氛围编程"。更准确地说，它是一种"凭感觉让 AI 写代码"的开发方式...

## 注意事项

- 如果用户没有提供原始 source URL，frontmatter 的 source 字段留空或省略
- 如果嘉宾身份不清楚，先问用户，或从 transcript 里推断
- 不要生成或编造不在 transcript 里的观点，即使加编者注也要注明来源
- 编辑手记里的批判要有事实依据，不能凭空怀疑
- 文章长度建议 2000-5000 字，根据 transcript 长度和信息密度决定
